## 1. 프로젝트 시작 전 확인사항

### 1.1 비즈니스 목표 정의

프로젝트를 시작하기 전에 다음 질문에 답할 수 있어야 합니다:

| 질문                         | 예시 답변                       |
| ---------------------------- | ------------------------------- |
| 해결하려는 비즈니스 문제는?  | 고객 이탈률이 15%로 높음        |
| 성공하면 어떤 가치가 있는가? | 이탈률 5%p 감소 시 연 10억 절감 |
| 결과를 어떻게 활용할 것인가? | 이탈 위험 고객에게 쿠폰 발송    |
| 언제까지 필요한가?           | 3개월 내                        |

### 1.2 데이터 준비 상태

| 항목                         | 확인                   |
| ---------------------------- | ---------------------- |
| 데이터 파일이 있는가?        | CSV, Excel, DB 등      |
| 데이터 접근이 가능한가?      | 권한, 보안 이슈        |
| 데이터 양은 충분한가?        | 최소 수천 건 이상 권장 |
| 타겟 변수가 정의되어 있는가? | 예측하려는 값          |

## 2. 분석 가설 수립 (핵심!)

> ⚠️ **가설 없이 EDA를 시작하면 방향 없이 데이터만 뒤지게 됩니다.**

### 2.1 좋은 가설의 조건

```
가설: [명확하고 검증 가능한 진술]
근거: [이 가설을 세운 이유]
검증방법: [EDA에서 어떻게 확인할지]
관련변수: [분석에 필요한 변수들]
예상결과: [가설이 맞다면 어떤 결과가 나올지]
```

### 2.2 가설 예시

**이탈 예측 프로젝트**

```
가설1: 계약 기간(tenure)이 24개월 미만인 고객은 이탈률이 높다
  - 근거: 신규 고객은 서비스 충성도가 낮음
  - 검증: 그룹별 이탈률 비교
  - 변수: tenure, churn
  - 예상: tenure < 24 이탈률 30% 이상

가설2: 월 요금(monthly_charges)이 높을수록 이탈률이 높다
  - 근거: 가격 민감도
  - 검증: 요금 구간별 이탈률, 상관관계
  - 변수: monthly_charges, churn
  - 예상: 상관계수 > 0.3

가설3: 고객센터 문의 횟수가 많은 고객은 이탈 가능성이 높다
  - 근거: 불만족 신호
  - 검증: 문의 횟수 그룹별 이탈률
  - 변수: support_calls, churn
  - 예상: 5회 이상 문의 고객 이탈률 40% 이상
```

## 3. 비즈니스 목표 → 분석 목표 변환

### 3.1 변환 패턴

| 비즈니스 목표 | 분석 목표        | 분석 유형     |
| ------------- | ---------------- | ------------- |
| 매출 증가     | 구매 확률 예측   | 분류          |
| 비용 감소     | 불량/고장 예측   | 분류/이상탐지 |
| 이탈 감소     | 이탈 확률 예측   | 분류          |
| 재고 최적화   | 수요 예측        | 시계열        |
| 마케팅 최적화 | 고객 세분화      | 군집          |
| 가격 최적화   | 가격 탄력성 분석 | 회귀          |

### 3.2 목표 설정 템플릿

```
비즈니스 목표: [달성하려는 것]
분석 대상: [어떤 데이터를 사용]
타겟 변수: [예측/분석하려는 값]
분석 유형: [분류/회귀/군집/시계열]
성공 기준: [정량적 지표와 목표값]
활용 방안: [모델 결과를 어떻게 쓸 것인지]
```

**예시**

```
비즈니스 목표: 고객 이탈률 5%p 감소
분석 대상: 고객 거래 내역 및 행동 데이터
타겟 변수: 30일 내 이탈 여부 (0/1)
분석 유형: 이진 분류
성공 기준: Recall ≥ 0.8 (이탈 고객 80% 이상 식별)
활용 방안: 이탈 위험 상위 20% 고객에게 리텐션 캠페인
```

## 4. 성공 기준 설정

### 4.1 분류 모델 지표 선택

| 상황                             | 권장 지표   | 이유               |
| -------------------------------- | ----------- | ------------------ |
| 클래스 균형                      | Accuracy    | 직관적 해석        |
| 클래스 불균형                    | F1, ROC-AUC | Accuracy 왜곡 방지 |
| 놓치면 안 되는 경우 (사기, 이탈) | Recall      | 탐지율 중시        |
| 오탐이 비용인 경우               | Precision   | 정확도 중시        |

### 4.2 회귀 모델 지표 선택

| 상황                       | 권장 지표 |
| -------------------------- | --------- |
| 일반적인 경우              | RMSE, R² |
| 이상치에 민감하면 안 될 때 | MAE       |
| 비율로 평가하고 싶을 때    | MAPE      |

### 4.3 현실적인 목표 설정

- 첫 모델 목표: 베이스라인 대비 10% 이상 개선
- 벤치마크 참고: 유사 프로젝트, 논문, 캐글 등
- 데이터 품질에 따라 조정: 결측치 많으면 목표 낮춤

## 8. 도메인별 분석 팁

### 마케팅/고객

- 타겟: 구매 여부, 이탈 여부, 반응 여부
- 주요 변수: RFM(최근성, 빈도, 금액), 인구통계, 채널
- 주의: 데이터 누수 (미래 정보 포함 금지)

### 제조/품질

- 타겟: 불량 여부, 고장 시점
- 주요 변수: 센서 데이터, 운전 조건, 환경
- 주의: 클래스 불균형 (불량률 1% 미만)

### 금융/리스크

- 타겟: 연체 여부, 사기 여부
- 주요 변수: 거래 패턴, 신용 정보
- 주의: 규제 준수, 설명 가능성

### 유통/수요예측

- 타겟: 판매량, 재고 소진 시점
- 주요 변수: 시계열, 프로모션, 계절성
- 주의: 외부 요인 (날씨, 이벤트)

## Git Commit Convention

커밋 시 Conventional Commits 형식 사용:

**절대 금지: 커밋 메시지에 Co-Authored-By: Claude 또는 유사 문구를 포함하지 않는다.**

- feat: 새 기능
- eda: 데이터 EDA
- pre : 데이터 전처리
- Model : 머신러닝 모델 설정
- data : data 수집
- fix: 버그 수정
- docs: 문서 수정
- style: 포맷팅, 세미콜론 등
- refactor: 리팩토링
- test: 테스트 추가/수정
- chore: 빌드, 설정 변경

형식: [권유석] `<type>`(`<scope>`): `<description>`

예시:

- [권유석]feat(auth): 로그인 기능 추가
- [권유석]fix(api): 응답 파싱 오류 수정
